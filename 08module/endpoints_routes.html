Anatomy of a REST API Endpoint

Endpoint Structure Breakdown

https://site.com/wp-json / wp / v2 / posts / 123

Components:

  • Base URL: https://site.com/wp-json - API root
  • Namespace: wp - Vendor/plugin identifier
  • Version: v2 - API version
  • Resource: posts - Type of content
  • Identifier: 123 - Specific resource ID

HTTP Methods in REST API

GET

Read/Retrieve

Fetch resources

Safe, Idempotent
POST

Create

Create new resources

Not Safe, Not Idempotent
PUT

Update/Replace

Full resource update

Not Safe, Idempotent
PATCH

Partial Update

Update specific fields

Not Safe, Not Idempotent
DELETE

Remove

Delete resources

Not Safe, Idempotent

Method Usage Examples

// GET - Retrieve all posts
fetch('/wp-json/wp/v2/posts')

// GET - Retrieve specific post
fetch('/wp-json/wp/v2/posts/123')

// POST - Create new post
fetch('/wp-json/wp/v2/posts', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'X-WP-Nonce': nonce
    },
    body: JSON.stringify({
        title: 'New Post',
        content: 'Post content',
        status: 'publish'
    })
})

// PUT - Update entire post
fetch('/wp-json/wp/v2/posts/123', {
    method: 'PUT',
    headers: {
        'Content-Type': 'application/json',
        'X-WP-Nonce': nonce
    },
    body: JSON.stringify({
        title: 'Updated Title',
        content: 'Updated content',
        status: 'publish'
    })
})

// DELETE - Remove post
fetch('/wp-json/wp/v2/posts/123', {
    method: 'DELETE',
    headers: {
        'X-WP-Nonce': nonce
    }
})

Resource Route Methods Description
Posts /wp/v2/posts GET, POST Blog posts collection
Post /wp/v2/posts/{id} GET, POST, PUT, DELETE Single post operations

Users /wp/v2/users GET, POST User accounts

Tags /wp/v2/tags GET, POST Post tags

Query Parameters and Filters

Common Query Parameters

Pagination Parameters

// Get 5 posts per page
/wp-json/wp/v2/posts?per_page=5

// Get page 2 of results
/wp-json/wp/v2/posts?per_page=5&page=2

// Offset results
/wp-json/wp/v2/posts?offset=10

Filtering Parameters

// Filter by author
/wp-json/wp/v2/posts?author=1

// Filter by category
/wp-json/wp/v2/posts?categories=5

// Filter by status
/wp-json/wp/v2/posts?status=draft

// Multiple filters
/wp-json/wp/v2/posts?author=1&categories=5&per_page=10

Ordering Parameters

// Order by date (default)
/wp-json/wp/v2/posts?orderby=date&order=desc

// Order by title
/wp-json/wp/v2/posts?orderby=title&order=asc

// Order by menu order
/wp-json/wp/v2/pages?orderby=menu_order

Search and Field Parameters

// Search posts
/wp-json/wp/v2/posts?search=wordpress

// Include specific fields only
/wp-json/wp/v2/posts?_fields=id,title,date

// Exclude fields
/wp-json/wp/v2/posts?_fields=author,id,excerpt,title,link

// Embed related resources
/wp-json/wp/v2/posts?_embed

Route Patterns and Regular Expressions

Common Route Patterns

Static Route

/myplugin/v1/settings

Fixed path with no variables

Numeric ID Parameter

/myplugin/v1/items/(?P<id>\d+)

Matches: /items/123

String Parameter

/myplugin/v1/categories/(?P<slug>[a-zA-Z0-9-]+)

Matches: /categories/my-category

Optional Parameter

/myplugin/v1/posts(?:/(?P<id>\d+))?

Matches: /posts or /posts/123

Multiple Parameters

/myplugin/v1/(?P<type>\w+)/(?P<status>\w+)

Matches: /posts/published

Route Discovery and Schema

Discovering Available Routes

// Discover all available routes
fetch('/wp-json')
    .then(response => response.json())
    .then(data => {
        console.log('Available namespaces:', data.namespaces);
        console.log('Available routes:', data.routes);
    });

// Get schema for specific route
fetch('/wp-json/wp/v2/posts', {
    method: 'OPTIONS'
})
    .then(response => response.json())
    .then(schema => {
        console.log('Endpoint schema:', schema);
        console.log('Supported methods:', schema.methods);
        console.log('Endpoint args:', schema.endpoints[0].args);
    });

üí° Discovery Tips:

  • Access /wp-json to see all routes
  • Use OPTIONS request to get endpoint schema
  • Check _links in responses for related routes
  • Use ?context=help for endpoint documentation

Pagination and Response Headers

Working with Paginated Responses

// Handle pagination headers
fetch('/wp-json/wp/v2/posts?per_page=10')
    .then(response => {
        // Get pagination info from headers
        const totalPages = response.headers.get('X-WP-TotalPages');
        const totalItems = response.headers.get('X-WP-Total');
        
        console.log(`Total items: ${totalItems}`);
        console.log(`Total pages: ${totalPages}`);
        
        // Get Link header for navigation
        const linkHeader = response.headers.get('Link');
        console.log('Navigation links:', linkHeader);
        
        return response.json();
    })
    .then(posts => {
        // Process posts
        console.log(`Received ${posts.length} posts`);
    });

Pagination Headers

  • X-WP-Total - Total number of items
  • X-WP-TotalPages - Total number of pages
  • Link - Navigation links (prev, next)

Working with Embedded Resources

Using _embed Parameter

// Get posts with embedded author and featured media
fetch('/wp-json/wp/v2/posts?_embed')
    .then(response => response.json())
    .then(posts => {
        posts.forEach(post => {
                        // Access embedded author
            const author = post._embedded.author[0];
            console.log('Author:', author.name);
            
            // Access embedded featured media
            if (post._embedded['wp:featuredmedia']) {
                const media = post._embedded['wp:featuredmedia'][0];
                console.log('Featured image:', media.source_url);
            }
            
            // Access embedded terms (categories, tags)
            if (post._embedded['wp:term']) {
                const categories = post._embedded['wp:term'][0];
                console.log('Categories:', categories);
            }
        });
    });

üí° Embedding Benefits:

  • Reduces number of API calls
  • Gets related data in single request
  • Improves performance
  • Simplifies frontend code

Best Practices for Routes and Endpoints

‚úÖ Do's

  • Use semantic, RESTful URLs
  • Follow naming conventions (plural for collections)
  • Use proper HTTP methods for operations
  • Include version in namespace
  • Validate and sanitize all parameters
  • Return appropriate status codes
  • Document your custom endpoints
  • Use consistent parameter naming

‚ùå Don'ts

  • Don't use verbs in endpoint URLs
  • Don't nest resources too deeply
  • Don't ignore HTTP method semantics
  • Don't return HTML from API endpoints
  • Don't expose sensitive data in URLs
  • Don't forget error handling
