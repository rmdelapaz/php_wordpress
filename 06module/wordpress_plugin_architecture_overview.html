What is a WordPress Plugin?

A WordPress plugin is a piece of software that extends or adds functionality to WordPress without modifying the core code.

Plugin Characteristics

  • Modular: Self-contained functionality
  • Extensible: Can be extended by other plugins
  • Portable: Works across WordPress installations
  • Upgradeable: Can be updated without affecting core
  • Deactivatable: Can be turned on/off without data loss
  • Customizable: Often includes settings and options

The WordPress Hook System

The hook system is the foundation of WordPress plugin architecture. It allows plugins to "hook into" WordPress at specific points during execution.

üéØ Actions

Actions allow you to add or change WordPress functionality.

  • Execute functions at specific points
  • Don't return values
  • Used for doing something
  • Example: sending email, saving data
add_action('hook_name', 'function_name');

üîÑ Filters

Filters allow you to modify data before it's used.

  • Modify and return values
  • Must return a value
  • Used for changing something
  • Example: modifying content, changing titles
add_filter('hook_name', 'function_name');

Action Hook Example

<?php
// Hook into WordPress initialization
add_action('init', 'my_plugin_init');

function my_plugin_init() {
    // This code runs when WordPress initializes
    register_post_type('product', array(
        'public' => true,
        'label' => 'Products'
    ));
}

// Hook with priority and arguments
add_action('save_post', 'my_save_post_function', 10, 2);

function my_save_post_function($post_id, $post) {
    // This runs when a post is saved
    // Do something with the post data
}

Filter Hook Example

<?php
// Modify the content before display
add_filter('the_content', 'my_content_filter');

function my_content_filter($content) {
    // Add something to the content
    $content .= '<p>Thanks for reading!</p>';
    return $content; // Must return the modified value
}

// Filter with priority and multiple arguments
add_filter('the_title', 'my_title_filter', 10, 2);

function my_title_filter($title, $id) {
    if (is_single()) {
        $title = 'üìñ ' . $title;
    }
    return $title;
}

Plugin Loading Order Example

<?php
// These hooks fire in order during WordPress loading

// Very early - when plugin file is included
register_activation_hook(__FILE__, 'my_plugin_activate');
register_deactivation_hook(__FILE__, 'my_plugin_deactivate');

// After all plugins are loaded
add_action('plugins_loaded', 'my_plugins_loaded');

// After pluggable functions
add_action('after_setup_theme', 'my_after_theme_setup');

// WordPress fully loaded but before headers sent
add_action('init', 'my_init_function');

// Admin area initialized
add_action('admin_init', 'my_admin_init');

// Before page content is sent
add_action('wp', 'my_wp_function');

// In the HTML head section
add_action('wp_head', 'my_head_function');

// Footer of the page
add_action('wp_footer', 'my_footer_function');

WordPress Plugin APIs

WordPress provides numerous APIs that plugins can use to extend functionality:

Core Plugin APIs

1. Options API

// Store and retrieve plugin settings
add_option('my_plugin_option', 'default_value');
$value = get_option('my_plugin_option');
update_option('my_plugin_option', 'new_value');
delete_option('my_plugin_option');

2. Database API

global $wpdb;
// Create custom tables
$wpdb->query("CREATE TABLE {$wpdb->prefix}my_table ...");
// Query data
$results = $wpdb->get_results("SELECT * FROM {$wpdb->prefix}my_table");

3. HTTP API

// Make external requests
$response = wp_remote_get('https://api.example.com/data');
$body = wp_remote_retrieve_body($response);

4. File System API

// Work with files
WP_Filesystem();
global $wp_filesystem;
$wp_filesystem->put_contents($file, $contents);

5. Settings API

// Create settings pages
register_setting('my_options_group', 'my_option_name');
add_settings_section('my_section', 'Section Title', 'callback', 'page');
add_settings_field('my_field', 'Field Title', 'callback', 'page', 'section');

6. Transients API

// Temporary data storage with expiration
set_transient('my_transient', $value, 3600); // 1 hour
$value = get_transient('my_transient');
delete_transient('my_transient');

Plugin Execution Context

Understanding Execution Context

<?php
/**
 * Plugins run in different contexts
 */

// Check if in admin area
if (is_admin()) {
    // Admin-only functionality
    add_action('admin_menu', 'add_admin_pages');
    add_action('admin_enqueue_scripts', 'enqueue_admin_scripts');
} else {
    // Frontend-only functionality
    add_action('wp_enqueue_scripts', 'enqueue_frontend_scripts');
}

// Check if doing AJAX
if (wp_doing_ajax()) {
    // AJAX handlers
    add_action('wp_ajax_my_action', 'handle_ajax');
    add_action('wp_ajax_nopriv_my_action', 'handle_ajax_nopriv');
}

// Check if in Customizer
if (is_customize_preview()) {
    // Customizer-specific code
    add_action('customize_register', 'register_customizer_settings');
}

// Check if doing cron
if (wp_doing_cron()) {
    // Scheduled tasks
    add_action('my_scheduled_event', 'do_scheduled_task');
}

// Check if REST API request
if (defined('REST_REQUEST') && REST_REQUEST) {
    // REST API specific code
    add_action('rest_api_init', 'register_rest_routes');
}

Security in Plugin Architecture

Security Best Practices

Nonce Verification

// Create nonce
$nonce = wp_create_nonce('my_action_nonce');

// Verify nonce
if (!wp_verify_nonce($_POST['_wpnonce'], 'my_action_nonce')) {
    die('Security check failed');
}

Capability Checks

// Check user permissions
if (!current_user_can('manage_options')) {
    wp_die('You do not have permission to access this page.');
}

Data Sanitization

// Sanitize input
$title = sanitize_text_field($_POST['title']);
$content = wp_kses_post($_POST['content']);
$email = sanitize_email($_POST['email']);

// Escape output
echo esc_html($title);
echo esc_url($url);
echo esc_attr($attribute);

Performance Considerations

Performance Best Practices

  • Load conditionally: Only load code when needed
  • Use caching: Cache expensive operations
  • Minimize database queries: Use efficient queries
  • Enqueue properly: Load scripts/styles correctly
  • Avoid globals: Use dependency injection
  • Lazy load: Load resources on demand
  • Use transients: Cache API calls and calculations
  • Optimize autoloading: Use efficient class loading

Performance Optimization Example

<?php
// Bad: Loading on every page
add_action('init', 'load_my_plugin_everything');

// Good: Load conditionally
add_action('init', 'load_my_plugin_conditionally');

function load_my_plugin_conditionally() {
    // Only load on specific pages
    if (is_page('contact')) {
        require_once plugin_dir_path(__FILE__) . 'contact-form.php';
    }
    
    // Only load in admin
    if (is_admin()) {
        require_once plugin_dir_path(__FILE__) . 'admin/admin.php';
    }
    
    // Cache expensive operation
    $data = get_transient('my_plugin_data');
    if (false === $data) {
        $data = expensive_operation();
        set_transient('my_plugin_data', $data, HOUR_IN_SECONDS);
    }
}

Common Architecture Pitfalls

Avoid these common mistakes that can break your plugin or cause conflicts

Pitfalls to Avoid

  1. Direct file access: Always check if WordPress is loaded
  2. Hardcoded paths: Use WordPress functions for paths
  3. Global namespace pollution: Prefix everything
  4. Direct database queries: Use $wpdb properly
  5. Loading order assumptions: Don't assume plugin order
  6. Missing dependency checks: Check if required plugins exist
  7. Improper hook usage: Wrong priority or wrong hook
  8. Not checking context: Admin vs frontend vs AJAX
