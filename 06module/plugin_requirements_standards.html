Meeting WordPress plugin development standards

Learn the requirements for creating quality, compliant plugins

Learning Objectives

  • Understand WordPress.org plugin requirements
  • Learn WordPress coding standards for plugins
  • Master security requirements
  • Understand licensing requirements
  • Learn documentation standards
  • Implement accessibility standards
  • Follow performance best practices
  • Prepare for plugin review process

Why Standards Matter

Following WordPress plugin standards ensures your plugin is secure, performant, maintainable, and compatible with the WordPress ecosystem.

Benefits of Following Standards

  • Compatibility: Works with different WordPress versions
  • Security: Protects users from vulnerabilities
  • Maintainability: Easier to update and debug
  • User Trust: Professional appearance and behavior
  • Repository Approval: Meet WordPress.org requirements
  • Community Support: Other developers can contribute

üî¥ Core Requirements (MUST HAVE)

  • Unique plugin name/slug REQUIRED
  • Valid plugin header REQUIRED
  • GPL-compatible license REQUIRED
  • No malicious or illegal code REQUIRED
  • Secure code (no vulnerabilities) REQUIRED
  • Proper data sanitization REQUIRED
  • Proper data escaping REQUIRED
  • No "powered by" or credit links REQUIRED
  • No obfuscated code REQUIRED
  • No external dependencies without fallback REQUIRED

PHP Coding Standards

Proper PHP Standards

<?php
/**
 * Plugin Name: My Awesome Plugin
 * Description: A properly formatted WordPress plugin
 * Version: 1.0.0
 * Author: Your Name
 */

// Exit if accessed directly
if ( ! defined( 'ABSPATH' ) ) {
    exit;
}

/**
 * Main plugin class
 * 
 * @since 1.0.0
 */
class My_Awesome_Plugin {
    
    /**
     * Plugin version
     * 
     * @var string
     */
    private $version = '1.0.0';
    
    /**
     * Constructor
     */
    public function __construct() {
        $this->define_constants();
        $this->load_dependencies();
        $this->set_hooks();
    }
    
    /**
     * Define plugin constants
     */
    private function define_constants() {
        define( 'MY_PLUGIN_VERSION', $this->version );
        define( 'MY_PLUGIN_PATH', plugin_dir_path( __FILE__ ) );
        define( 'MY_PLUGIN_URL', plugin_dir_url( __FILE__ ) );
    }
    
    /**
     * Load required files
     */
    private function load_dependencies() {
        require_once MY_PLUGIN_PATH . 'includes/class-loader.php';
        require_once MY_PLUGIN_PATH . 'includes/functions.php';
    }
    
    /**
     * Register hooks
     */
    private function set_hooks() {
        add_action( 'init', array( $this, 'init' ) );
        add_action( 'wp_enqueue_scripts', array( $this, 'enqueue_scripts' ) );
    }
    
    /**
     * Initialize plugin
     */
    public function init() {
        // Plugin initialization code
        load_plugin_textdomain( 'my-plugin', false, dirname( plugin_basename( __FILE__ ) ) . '/languages' );
    }
    
    /**
     * Enqueue scripts and styles
     */
    public function enqueue_scripts() {
        wp_enqueue_style( 
            'my-plugin-style', 
            MY_PLUGIN_URL . 'assets/css/style.css', 
            array(), 
            MY_PLUGIN_VERSION 
        );
        
        wp_enqueue_script( 
            'my-plugin-script', 
            MY_PLUGIN_URL . 'assets/js/script.js', 
            array( 'jquery' ), 
            MY_PLUGIN_VERSION, 
            true 
        );
    }
}

// Initialize the plugin
new My_Awesome_Plugin();

Key PHP Standards:

  • Use tabs for indentation (not spaces)
  • Opening braces on same line for control structures
  • Space after control structure keywords
  • Yoda conditions for comparisons
  • Proper naming conventions (snake_case for functions)
  • DocBlocks for all functions and classes

Security Requirements

üîí Security Standards

Data Sanitization

<?php
// Sanitize text input
$title = sanitize_text_field( $_POST['title'] );

// Sanitize textarea
$description = sanitize_textarea_field( $_POST['description'] );

// Sanitize email
$email = sanitize_email( $_POST['email'] );

// Sanitize URL
$website = esc_url_raw( $_POST['website'] );

// Sanitize HTML
$content = wp_kses_post( $_POST['content'] );

// Sanitize SQL
global $wpdb;
$query = $wpdb->prepare( 
    "SELECT * FROM {$wpdb->posts} WHERE ID = %d", 
    $post_id 
);

Data Escaping

<?php
// Escape HTML
echo esc_html( $title );

// Escape attributes
echo '<input type="text" value="' . esc_attr( $value ) . '">';

// Escape URLs
echo '<a href="' . esc_url( $link ) . '">Link</a>';

// Escape JavaScript
echo '<script>var data = ' . wp_json_encode( $data ) . ';</script>';

// Escape SQL
$safe_sql = esc_sql( $user_input );

Nonce Verification

<?php
// Create nonce
$nonce = wp_create_nonce( 'my_action_nonce' );

// Add to form
echo '<input type="hidden" name="_wpnonce" value="' . esc_attr( $nonce ) . '">';

// Verify nonce
if ( ! isset( $_POST['_wpnonce'] ) || ! wp_verify_nonce( $_POST['_wpnonce'], 'my_action_nonce' ) ) {
    die( 'Security check failed' );
}

// AJAX nonce
wp_localize_script( 'my-script', 'my_ajax', array(
    'nonce' => wp_create_nonce( 'my_ajax_nonce' ),
    'url'   => admin_url( 'admin-ajax.php' )
) );

Capability Checks

<?php
// Check user capabilities
if ( ! current_user_can( 'manage_options' ) ) {
    wp_die( __( 'You do not have sufficient permissions to access this page.' ) );
}

// Check specific capability
if ( ! current_user_can( 'edit_post', $post_id ) ) {
    return;
}

// Check multiple capabilities
if ( current_user_can( 'edit_posts' ) || current_user_can( 'edit_pages' ) ) {
    // User can edit content
}

Naming Conventions

Element Convention Example
Plugin Folder lowercase-with-hyphens my-awesome-plugin
Main Plugin File Same as folder my-awesome-plugin.php
Functions Prefixed snake_case my_plugin_function_name()
Classes Prefixed Pascal_Case My_Plugin_Class_Name
Constants Prefixed UPPERCASE MY_PLUGIN_CONSTANT
Hooks Prefixed snake_case my_plugin_custom_hook
Text Domain Same as plugin folder my-awesome-plugin
Options Prefixed snake_case my_plugin_settings
Always prefix everything! This prevents conflicts with other plugins, themes, and WordPress core.

PHPDoc Standards

<?php
/**
 * Plugin main file
 *
 * @package     My_Awesome_Plugin
 * @author      Your Name
 * @copyright   2024 Your Company
 * @license     GPL-2.0+
 *
 * @wordpress-plugin
 * Plugin Name: My Awesome Plugin
 * Plugin URI:  https://example.com/plugin
 * Description: This plugin does awesome things
 * Version:     1.0.0
 * Author:      Your Name
 * Author URI:  https://example.com
 * Text Domain: my-awesome-plugin
 * Domain Path: /languages
 * License:     GPL-2.0+
 * License URI: http://www.gnu.org/licenses/gpl-2.0.txt
 */

/**
 * Register custom post type
 *
 * Creates a new post type for products with custom capabilities
 * and taxonomies.
 *
 * @since 1.0.0
 * @access public
 * 
 * @param array $args {
 *     Optional. Array of arguments.
 *     
 *     @type string $label      Post type label. Default 'Products'.
 *     @type bool   $public     Whether public. Default true.
 *     @type array  $supports   Post type supports. Default title, editor.
 * }
 * @return void
 */
function my_plugin_register_post_type( $args = array() ) {
    $defaults = array(
        'label'    => __( 'Products', 'my-awesome-plugin' ),
        'public'   => true,
        'supports' => array( 'title', 'editor' )
    );
    
    $args = wp_parse_args( $args, $defaults );
    
    register_post_type( 'product', $args );
}

WordPress.org Repository Requirements

üì¶ Repository Specific Requirements

  • Unique plugin name not in use
  • Detailed readme.txt file
  • Stable tag in readme.txt
  • License explicitly stated
  • No external service requirements
  • No "phoning home" without consent
  • No advertising in admin
  • Trialware not allowed
  • Service integrations must be useful without service
  • Must use WordPress functions when available

Performance Standards

Performance Best Practices

  • Conditional Loading: Only load assets when needed
  • Database Optimization: Use proper indexes and caching
  • Minimize HTTP Requests: Combine and minify assets
  • Use WordPress APIs: Leverage built-in caching
  • Lazy Loading: Load resources on demand
  • Efficient Queries: Avoid slow database queries
  • Asset Versioning: Enable browser caching
  • Background Processing: Use wp-cron for heavy tasks

Conditional Asset Loading

<?php
// Only load on specific pages
add_action( 'wp_enqueue_scripts', 'my_plugin_conditional_scripts' );

function my_plugin_conditional_scripts() {
    // Only on single posts
    if ( is_single() ) {
        wp_enqueue_script( 'my-single-script', plugin_dir_url( __FILE__ ) . 'js/single.js' );
    }
    
    // Only on pages with shortcode
    global $post;
    if ( is_a( $post, 'WP_Post' ) && has_shortcode( $post->post_content, 'my_shortcode' ) ) {
        wp_enqueue_style( 'my-shortcode-style', plugin_dir_url( __FILE__ ) . 'css/shortcode.css' );
    }
    
    // Only on specific page template
    if ( is_page_template( 'template-contact.php' ) ) {
        wp_enqueue_script( 'my-contact-script', plugin_dir_url( __FILE__ ) . 'js/contact.js' );
    }
}

Testing Requirements

Pre-Release Testing Checklist

  • Plugin activates without errors
  • Plugin deactivates cleanly
  • No PHP errors with WP_DEBUG enabled
  • No JavaScript console errors
  • Works with latest WordPress version
  • Compatible with PHP 7.4+
  • Mobile responsive
  • Cross-browser testing
  • Multisite compatible (if applicable)
  • No conflicts with popular plugins
  • Security scanning passed
  • Performance testing completed
  • Accessibility testing done
  • Internationalization ready
  • Documentation complete

Common Standards Violations

These violations will cause plugin rejection or security issues

Avoid These Mistakes

  1. Direct database queries without $wpdb
  2. Using $_REQUEST, $_GET, $_POST without sanitization
  3. Echoing variables without escaping
  4. Including external scripts from CDNs
  5. Not checking user capabilities
  6. Using generic function names without prefix
  7. Hardcoding paths instead of using constants
  8. Not using nonces for form submissions
  9. Loading scripts/styles on all pages
  10. Storing sensitive data in plain text
