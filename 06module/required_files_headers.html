Plugin File Requirements

WordPress plugins have specific file requirements and conventions. Understanding these ensures your plugin is recognized, properly documented, and maintainable.

File Categories

  • Required Files: Absolutely necessary for plugin to function
  • Recommended Files: Best practice to include
  • Optional Files: Enhance functionality or documentation
  • Repository Files: Required for WordPress.org submission

The Plugin Header

The plugin header is a PHP comment block at the top of your main plugin file that tells WordPress about your plugin.

Complete Plugin Header Example

<?php
/**
 * Plugin Name:       My Awesome Plugin
 * Plugin URI:        https://example.com/plugins/my-awesome-plugin/
 * Description:       This plugin adds awesome functionality to your WordPress site.
 * Version:           1.2.3
 * Requires at least: 5.2
 * Requires PHP:      7.2
 * Author:            John Doe
 * Author URI:        https://example.com/
 * License:           GPL v2 or later
 * License URI:       https://www.gnu.org/licenses/gpl-2.0.html
 * Update URI:        https://example.com/my-plugin-updates/
 * Text Domain:       my-plugin
 * Domain Path:       /languages
 * Network:           true
 */

// Plugin code starts here...

Header Fields in Detail

Plugin Name Required

The display name of your plugin. This appears in the plugins list.

Plugin Name: My Awesome Plugin
  • Keep it concise and descriptive
  • Avoid trademark violations
  • Use title case
  • Maximum 200 characters

Version Recommended

Current version using semantic versioning (MAJOR.MINOR.PATCH)

Version: 1.2.3
  • MAJOR: Breaking changes
  • MINOR: New features, backward compatible
  • PATCH: Bug fixes
  • Used for update notifications

Text Domain Recommended

Unique identifier for translation strings

Text Domain: my-plugin
  • Must match plugin folder name
  • Use lowercase and hyphens
  • No spaces or special characters
  • Used in translation functions

Main Plugin File Structure

Complete Main Plugin File Template

<?php
/**
 * Plugin Name: My Plugin
 * Description: A well-structured WordPress plugin
 * Version: 1.0.0
 * Author: Your Name
 * License: GPL v2 or later
 * Text Domain: my-plugin
 */

// Exit if accessed directly
if ( ! defined( 'ABSPATH' ) ) {
    exit; // Exit if accessed directly
}

// Define plugin constants
define( 'MY_PLUGIN_VERSION', '1.0.0' );
define( 'MY_PLUGIN_FILE', __FILE__ );
define( 'MY_PLUGIN_PATH', plugin_dir_path( __FILE__ ) );
define( 'MY_PLUGIN_URL', plugin_dir_url( __FILE__ ) );
define( 'MY_PLUGIN_BASENAME', plugin_basename( __FILE__ ) );

// Require the main plugin class
require_once MY_PLUGIN_PATH . 'includes/class-my-plugin.php';

// Initialize the plugin
function my_plugin_init() {
    $plugin = new My_Plugin();
    $plugin->run();
}
add_action( 'plugins_loaded', 'my_plugin_init' );

// Activation hook
register_activation_hook( __FILE__, 'my_plugin_activate' );
function my_plugin_activate() {
    require_once MY_PLUGIN_PATH . 'includes/class-my-plugin-activator.php';
    My_Plugin_Activator::activate();
}

// Deactivation hook
register_deactivation_hook( __FILE__, 'my_plugin_deactivate' );
function my_plugin_deactivate() {
    require_once MY_PLUGIN_PATH . 'includes/class-my-plugin-deactivator.php';
    My_Plugin_Deactivator::deactivate();
}

The readme.txt File

Required for WordPress.org repository submission. Provides detailed information about your plugin.

readme.txt Template

=== My Plugin ===
Contributors: yourusername
Donate link: https://example.com/donate
Tags: tag1, tag2, tag3
Requires at least: 5.2
Tested up to: 6.4
Stable tag: 1.0.0
Requires PHP: 7.2
License: GPLv2 or later
License URI: https://www.gnu.org/licenses/gpl-2.0.html

Short description of your plugin, no more than 150 characters.

== Description ==

This is the long description. No limit, and you can use Markdown (as well as in the following sections).

For backwards compatibility, if this section is missing, the full length of the short description will be used, and
Markdown parsed.

A few notes about the sections above:

* "Contributors" is a comma separated list of wordpress.org usernames
* "Tags" is a comma separated list of tags that apply to the plugin
* "Requires at least" is the lowest version that the plugin will work on
* "Tested up to" is the highest version that you've *successfully used to test the plugin*

== Installation ==

1. Upload `my-plugin` folder to the `/wp-content/plugins/` directory
2. Activate the plugin through the 'Plugins' menu in WordPress
3. Go to Settings > My Plugin to configure

== Frequently Asked Questions ==

= How do I configure the plugin? =

Navigate to Settings > My Plugin in your WordPress admin area.

= Is this plugin compatible with my theme? =

This plugin should work with any properly coded WordPress theme.

== Screenshots ==

1. This screen shot description corresponds to screenshot-1.(png|jpg|jpeg|gif).
2. This is the second screen shot description.

== Changelog ==

= 1.0.0 =
* Initial release
* Added feature X
* Added feature Y

= 0.9.0 =
* Beta release

== Upgrade Notice ==

= 1.0.0 =
This version fixes a security issue. Upgrade immediately.

== Additional Info ==

For more information, visit [plugin homepage](https://example.com).

The uninstall.php File

Executes when the plugin is deleted through the WordPress admin.

uninstall.php Template

<?php
/**
 * Fired when the plugin is uninstalled.
 *
 * When populating this file, consider the following flow
 * of control:
 *
 * - This method should be static
 * - Check if the $_REQUEST content actually is the plugin name
 * - Run an admin referrer check to make sure it goes through authentication
 * - Verify the output of $_GET makes sense
 * - Repeat with other user roles. Best directly by using the links/query string parameters.
 * - Repeat things for multisite. Once for a single site in the network, once sitewide.
 *
 * @link       https://example.com
 * @since      1.0.0
 *
 * @package    My_Plugin
 */

// If uninstall not called from WordPress, then exit.
if ( ! defined( 'WP_UNINSTALL_PLUGIN' ) ) {
    exit;
}

// Clear database stored data
$options = array(
    'my_plugin_option_1',
    'my_plugin_option_2',
    'my_plugin_settings',
);

foreach ( $options as $option ) {
    delete_option( $option );
}

// Delete custom database tables
global $wpdb;
$table_name = $wpdb->prefix . 'my_plugin_table';
$wpdb->query( "DROP TABLE IF EXISTS $table_name" );

// Delete user meta
$wpdb->query( "DELETE FROM $wpdb->usermeta WHERE meta_key LIKE 'my_plugin_%'" );

// Delete post meta
$wpdb->query( "DELETE FROM $wpdb->postmeta WHERE meta_key LIKE '_my_plugin_%'" );

// Clear any cached data that has been removed
wp_cache_flush();
Be careful with uninstall.php! It permanently deletes data and cannot be undone.

Security Files

index.php (Directory Protection)

<?php
// Silence is golden.

Place this file in every directory to prevent directory browsing.

.htaccess (Additional Protection)

# Protect files
<FilesMatch "\.(php|inc)$">
    Order allow,deny
    Deny from all
</FilesMatch>

# Allow main plugin file
<Files "my-plugin.php">
    Order allow,deny
    Allow from all
</Files>

File and Header Best Practices

  • Consistent naming: Use same slug everywhere
  • Version control: Track all file changes
  • Documentation: Keep files well-documented
  • Security first: Include index.php files
  • Clean uninstall: Remove all plugin data
  • Proper licensing: Be clear about terms
  • Regular updates: Keep headers current
  • Translation ready: Include .pot file
